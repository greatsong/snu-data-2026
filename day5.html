발표, 그리고 더 멋진 프로젝트로

오늘의 약속 — 5일 동안 만든 걸 세상에 보여 주고, 앞으로 더 갈 수 있는 지도 세 장을 챙겨 갑니다. 새 코드를 외우는 날이 아니에요.

🎯 오늘 하는 것
  • 프로젝트 마무리 — 내 앱을 발표할 수 있게 다듬고 공용 보드에 게시
  • 갤러리 워크 발표 — 서로 돌아보고, 좋아요·질문을 주고받기
  • 더 멀리 가는 꿀팁 3개 — 진짜 DB · 자동화 + 재미 API · 다음 걸음

1. 프로젝트 마무리 — 발표할 수 있게

거창하게 고칠 필요 없어요. 발표에서 자랑할 딱 세 가지만 또렷하면 됩니다. 지금부터 코칭 시간에 이 세 가지를 다듬어요.

발표에서 말할 것한 줄로 준비
무슨 질문내가 던진 질문 (예: “서울은 100년 동안 얼마나 더워졌나?”)
무슨 데이터어디서 온 데이터인지 (예: 기상청 기온, 구글시트 설문)
무슨 발견보고 나서 알게 된 것 한 가지 (예: “여름이 3주 길어졌다”)
  1. 앱 URL을 공용 보드에 게시
    5일 내내 써 온 공용 보드(구글시트 링크판 또는 패들렛)에 내 앱 주소를 붙입니다. 이 보드가 오늘 발표 무대가 돼요. 주소 옆에 이름과 한 줄 제목도 같이.
  2. 발표 1장 요약 만들기
    위 표의 질문 · 데이터 · 발견을 그대로 세 줄로 적으면 그게 발표 대본이에요. 화면에는 내 앱을 띄우고, 이 세 줄만 말하면 충분합니다.
💡 완성도보다 이야기

버튼 하나가 덜 예뻐도 괜찮아요. 발표의 주인공은 “내가 무엇이 궁금해서, 무엇을 발견했는지”예요. 미완성이어도 그 이야기가 있으면 훌륭한 발표입니다.

2. 발표하는 법 — 갤러리 워크

모두가 무대에 서면 시간이 모자라요. 그래서 미술관을 걷듯 서로의 작품을 돌아보는 방식으로 합니다. 순서는 이래요.

  1. 전원 URL 게시
    공용 보드에 모두의 앱 주소가 모입니다. 이게 오늘의 전시장이에요.
  2. 자유롭게 돌며 체험
    다른 사람 앱을 직접 열어 눌러 보세요. 마음에 들면 ‘좋아요’, 좋았던 점은 한 줄 피드백으로 남깁니다. (보드에 댓글/스티커로)
  3. 하이라이트 무대
    그중 몇 명이 앞에 나와 2~3분씩 자기 앱을 소개합니다. 질문·데이터·발견 세 줄이면 끝.
  4. ‘의심하는 눈’으로 질문 1개
    발표를 들으면 질문 하나씩 던져 봐요. “이 숫자, 다른 해랑 비교하면?”, “이 데이터는 언제 걸까?” 같은 질문이 이 강의에서 가장 값진 습관이에요.
✔️ ‘의심하는 눈’이 왜 중요할까요?

데이터를 그냥 믿지 않고 “정말일까? 어디서 왔을까?” 되묻는 것 — 이게 데이터 분석의 진짜 실력이에요. 서로의 발표에 질문을 던지는 연습이 곧 내 프로젝트를 보는 눈이 됩니다.

3. [꿀팁 1] 진짜 데이터베이스 — 구글시트를 넘어설 때

지금까지는 구글시트에 데이터를 쌓았죠. 편하지만, 사람이 많이 몰리거나 데이터가 수만 줄이 되면 느려지고 엉켜요. 그때 데이터베이스(DB)로 갈아탑니다. 오늘은 구경만 해요 — 필요할 때 꺼내 쓰는 서랍입니다.

🤔 언제 시트 대신 DB?

① 여러 사람이 동시에 쓰고 저장할 때 · ② 데이터가 아주 많을 때(수만 줄~) · ③ “요일별 평균”처럼 집계·계산을 빠르게 하고 싶을 때. 그 전까지는 구글시트로 충분합니다.

Supabase — 클릭으로 만드는 SQL 데이터베이스

Supabase는 무료로 시작하는 데이터베이스예요. 웹에서 표(테이블)를 만들고, 파이썬 도구 supabase-py로 데이터를 넣고(.insert()) 꺼냅니다(.select()).

② 개념 예시 (파이썬)
from supabase import create_client

# 프로젝트 주소와 키는 Supabase 대시보드에서 복사
supabase = create_client(URL, KEY)

# 데이터 한 줄 넣기
supabase.table("visits").insert({"city": "서울", "temp": 31.2}).execute()

# 데이터 꺼내기 (서울 것만)
rows = supabase.table("visits").select("*").eq("city", "서울").execute()
💡 RLS — DB의 잠금장치

Supabase에는 RLS(Row Level Security, 행 수준 보안)가 있어요. 기본은 전부 차단이고, “이건 읽어도 돼”처럼 허용 규칙(정책)을 켠 것만 보입니다. 그래서 처음엔 “데이터가 안 보여요” 하는데, 그건 고장이 아니라 안전하게 잠겨 있는 거예요.

🔑 키 두 종류 — 절대 헷갈리지 마세요

public(anon) 키는 앱에 넣어도 되는 공개용 열쇠(잠금장치 RLS가 지켜줌). service_role 키는 모든 잠금을 여는 마스터 열쇠라, 웹앱이나 GitHub에 절대 올리면 안 돼요. 비밀 금고(Secrets)에만 보관합니다.

Neon — SQL을 직접 쓰는 PostgreSQL

Neon은 PostgreSQL(대표적인 SQL 데이터베이스)을 무료로 켜 주는 서비스예요. 스트림릿에서는 st.connection(type="sql") 한 줄로 연결하고, SQL 문장을 직접 써서 데이터를 뽑습니다.

② 개념 예시 (스트림릿 + SQL)
import streamlit as st

conn = st.connection("neon", type="sql")   # Neon 연결

# 도시별로 방문 수를 세어 많은 순으로 상위 5개
df = conn.query("""
    SELECT city, COUNT(*) AS n
    FROM visits
    GROUP BY city
    ORDER BY n DESC
    LIMIT 5
""")
st.bar_chart(df, x="city", y="n")
💡 SQL 단어 네 개면 집계가 된다

COUNT(세기) · GROUP BY(묶기) · ORDER BY(정렬) · LIMIT(개수 제한). “도시별로 묶어(GROUP BY) 세고(COUNT), 많은 순으로(ORDER BY) 5개만(LIMIT)” — 이렇게 읽으면 됩니다. 물론 이 SQL도 AI에게 시켜서 받으면 돼요.

4. [꿀팁 2] 자동화 + 재미있는 API

데이터를 사람이 매일 손으로 모으면 지쳐요. 컴퓨터가 정해진 시간마다 알아서 모으게 하면? 그게 자동화예요. 그리고 세상엔 재미있는 데이터를 공짜로 주는 API가 아주 많습니다.

GitHub Actions — 매시간 알아서 데이터 쌓기

GitHub Actions는 GitHub이 정해진 시간마다 내 코드를 대신 실행해 주는 기능이에요. 저장소에 .github/workflows/ 폴더를 만들고 *.yml 파일 하나를 두면 됩니다.

② 개념 예시 (.github/workflows/collect.yml)
name: 데이터 수집
on:
  schedule:
    - cron: "5 * * * *"      # 매시간 5분에 실행 (UTC 기준!)
  workflow_dispatch:          # 이 줄이 있으면 '수동 실행' 버튼도 생김
jobs:
  run:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: python collect.py
        env:
          API_KEY: ${{ secrets.API_KEY }}   # 비밀키는 Secrets에서
⏰ cron 시간은 UTC예요 (한국시간 아님)

cron: "5 * * * *"는 ‘매시간 5분에’라는 뜻인데, 기준이 UTC(세계표준시)예요. 한국시간은 여기에 +9시간. 그래서 “한국 낮 12시에 돌리고 싶다”면 UTC로는 새벽 3시(0 3 * * *)로 적어야 합니다.

💡 비밀키는 GitHub Secrets에

API 키 같은 비밀은 코드에 직접 쓰지 말고 저장소 Settings → Secrets에 넣어요. 위 예시의 ${{ secrets.API_KEY }}가 그걸 꺼내 쓰는 방식입니다. workflow_dispatch를 넣어 두면 Actions 탭의 수동 버튼으로 즉시 시험해 볼 수 있어요.

💡 API도 결국 세 박자

“이 API로 이런 화면을 만들어 줘. 키는 여기 있어”라고 AI에게 주문하면 됩니다. 우리는 어떤 API가 있는지 서랍만 알면 돼요. 나머지는 1일차부터 배운 ① 주문 → ② 확인 → ③ 개선 그대로.

🌱 더 배우고 싶다면

지금처럼 “궁금한 질문 → 데이터 찾기 → AI에게 앱 주문 → 배포 → 의심하며 다듬기”를 반복하면 됩니다. 새 도구(진짜 DB, 자동화, 재미 API)는 필요해질 때 하나씩 서랍에서 꺼내면 돼요. 오늘 배운 건 도구가 아니라 이 리듬이에요.

🎒 오늘의 체크 & 마무리

  • 내 앱 URL을 공용 보드에 게시했다
  • 발표 세 줄(질문·데이터·발견)을 준비했다
  • 다른 사람 앱을 체험하고 좋아요·피드백을 남겼다
  • 진짜 DB / 자동화 / 재미 API가 ‘필요할 때 여는 서랍’이라는 걸 안다
  • 데이터를 볼 때 ‘의심하는 눈’으로 질문할 줄 안다
