발표, 그리고 더 멋진 프로젝트로

오늘 = 만든 걸 보여 주기 + 앞으로 갈 지도 세 장 챙기기. 새 코드 외우는 날 아님.

🎯 오늘 하는 것
  • 프로젝트 마무리 — 발표용으로 다듬고 공용 보드에 게시
  • 갤러리 워크 발표 — 서로 돌아보고 좋아요·질문
  • 꿀팁 3개 — 진짜 DB · 자동화 + 재미 API · 다음 걸음

1. 프로젝트 마무리 — 발표할 수 있게

  • 거창하게 고칠 필요 없음
  • 발표에서 자랑할 딱 세 가지만 또렷하게 ↓
발표에서 말할 것한 줄로 준비
무슨 질문예) “서울은 100년 동안 얼마나 더워졌나?”
무슨 데이터예) 기상청 기온, 구글시트 설문
무슨 발견예) “여름이 3주 길어졌다”
  1. 앱 URL을 공용 보드에 게시
    공용 보드(구글시트 링크판·패들렛)에 앱 주소 + 이름 + 한 줄 제목. 이 보드 = 오늘의 발표 무대
  2. 발표 1장 요약 만들기
    질문·데이터·발견 세 줄 = 발표 대본. 화면엔 내 앱, 세 줄만 말하면 충분
💡 완성도보다 이야기
  • 버튼 덜 예뻐도 OK
  • 주인공 = “무엇이 궁금해서, 무엇을 발견했나”
  • 미완성이어도 이야기 있으면 훌륭한 발표

2. 발표하는 법 — 갤러리 워크

  • 전원 무대 = 시간 부족 → 미술관 걷듯 서로 관람
  1. 전원 URL 게시
    공용 보드에 모두의 앱 주소 집결 = 오늘의 전시장
  2. 자유롭게 돌며 체험
    다른 앱 직접 눌러 보기 → 좋아요 + 한 줄 피드백(댓글/스티커)
  3. 하이라이트 무대
    몇 명이 앞에서 2~3분 소개. 질문·데이터·발견 세 줄이면 끝
  4. ‘의심하는 눈’으로 질문 1개
    예) “이 숫자, 다른 해랑 비교하면?” “이 데이터는 언제 걸까?” → 가장 값진 습관
✔️ ‘의심하는 눈’이 왜 중요?
  • 그냥 믿지 않고 “정말일까? 어디서 왔을까?” 되묻기
  • = 데이터 분석의 진짜 실력
  • 서로 질문 던지기 → 내 프로젝트 보는 눈

3. [꿀팁 1] 진짜 데이터베이스 — 구글시트를 넘어설 때

  • 지금까지 = 구글시트에 저장 (편하지만 대량·동시엔 느려짐)
  • 그때 = 데이터베이스(DB)로 전환 · 오늘은 구경만
🤔 언제 시트 대신 DB?
  • ① 여러 사람 동시 저장
  • ② 데이터 아주 많을 때(수만 줄~)
  • ③ 집계·계산을 빠르게 (예: 요일별 평균)
  • 그 전까지 = 구글시트로 충분

Supabase — 클릭으로 만드는 SQL 데이터베이스

  • 무료로 시작하는 DB
  • 웹에서 표(테이블) 생성 → supabase-py로 넣기(.insert())·꺼내기(.select())
② 개념 예시 (파이썬)
from supabase import create_client

# 프로젝트 주소와 키는 Supabase 대시보드에서 복사
supabase = create_client(URL, KEY)

# 데이터 한 줄 넣기
supabase.table("visits").insert({"city": "서울", "temp": 31.2}).execute()

# 데이터 꺼내기 (서울 것만)
rows = supabase.table("visits").select("*").eq("city", "서울").execute()
💡 RLS — DB의 잠금장치
  • RLS(Row Level Security) = 기본 전부 차단
  • 허용 규칙(정책) 켠 것만 보임
  • “데이터 안 보여요” = 고장 아님, 안전하게 잠긴 것
🔑 키 두 종류 — 절대 헷갈리지 말 것
  • public(anon) 키 = 앱에 넣어도 되는 공개용 (RLS가 보호)
  • service_role 키 = 모든 잠금 여는 마스터 → 웹앱·GitHub 업로드 금지
  • 비밀 금고(Secrets)에만 보관

Neon — SQL을 직접 쓰는 PostgreSQL

  • PostgreSQL을 무료로 켜 주는 서비스
  • 스트림릿 → st.connection(type="sql") 한 줄 연결 + SQL 직접 작성
② 개념 예시 (스트림릿 + SQL)
import streamlit as st

conn = st.connection("neon", type="sql")   # Neon 연결

# 도시별로 방문 수를 세어 많은 순으로 상위 5개
df = conn.query("""
    SELECT city, COUNT(*) AS n
    FROM visits
    GROUP BY city
    ORDER BY n DESC
    LIMIT 5
""")
st.bar_chart(df, x="city", y="n")
💡 SQL 단어 네 개면 집계 끝
  • COUNT(세기) · GROUP BY(묶기) · ORDER BY(정렬) · LIMIT(개수)
  • “도시별로 묶어 세고, 많은 순으로 5개만”
  • 이 SQL도 AI에게 시켜서 받으면 됨

4. [꿀팁 2] 자동화 + 재미있는 API

  • 사람이 매일 손으로 = 지침 → 컴퓨터가 정해진 시간마다 = 자동화
  • 세상엔 공짜 재미 API 많음

GitHub Actions — 매시간 알아서 데이터 쌓기

  • GitHub이 정해진 시간마다 내 코드 대신 실행
  • .github/workflows/에 *.yml 하나 두면 됨
② 개념 예시 (.github/workflows/collect.yml)
name: 데이터 수집
on:
  schedule:
    - cron: "5 * * * *"      # 매시간 5분에 실행 (UTC 기준!)
  workflow_dispatch:          # 이 줄이 있으면 '수동 실행' 버튼도 생김
jobs:
  run:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: python collect.py
        env:
          API_KEY: ${{ secrets.API_KEY }}   # 비밀키는 Secrets에서
⏰ cron 시간은 UTC (한국시간 아님)
  • cron: "5 * * * *" = 매시간 5분에
  • 기준 = UTC · 한국시간 = +9시간
  • 한국 낮 12시 → UTC 새벽 3시(0 3 * * *)
💡 비밀키는 GitHub Secrets에
  • API 키 = 코드 직접 작성 금지 → Settings → Secrets
  • 꺼내 쓰기 = ${{ secrets.API_KEY }}
  • workflow_dispatch → Actions 탭 수동 버튼으로 즉시 시험

재미있는 API 서랍

  • “언젠가 만들 수 있구나” 열어 두는 서랍 · 지금 다 안 해도 됨

💡 API도 결국 세 박자
  • “이 API로 이런 화면 만들어 줘, 키는 여기” → AI에 주문
  • 우리는 어떤 API 있는지 서랍만 알면 됨
  • 나머지 = ① 주문 → ② 확인 → ③ 개선

🎒 오늘의 체크 & 마무리

  • 내 앱 URL을 공용 보드에 게시했다
  • 발표 세 줄(질문·데이터·발견)을 준비했다
  • 다른 사람 앱을 체험하고 좋아요·피드백을 남겼다
  • 진짜 DB / 자동화 / 재미 API = ‘필요할 때 여는 서랍’
  • 데이터를 ‘의심하는 눈’으로 질문할 줄 안다
