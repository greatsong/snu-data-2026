프로젝트 발표와 심화 — 강사 진행안

학생용 5일차 페이지 ↗ 를 화면에 띄운 채 진행합니다. 오늘은 ‘가르치는 날’이 아니라 ‘보여 주고 배웅하는 날’. 발표 운영과 “서랍 소개”가 전부입니다.

🎙️ 오늘의 목표 선언 (개강 멘트)

“오늘은 새로 외우는 게 없어요. 5일 동안 만든 걸 서로에게 보여 주고, 앞으로 더 갈 지도 세 장을 챙겨 갑니다. 마지막 꿀팁들은 지금 다 안 해도 되는 서랍이에요.”

📖 오늘의 설계 원칙 — 전원 참여 발표

완전 입문 40명. 전원 무대는 시간상 불가하고, 발표 부담도 큽니다. 그래서 갤러리 워크(전원 참여) + 하이라이트 발표(화면 공유)(소수) 하이브리드로 갑니다. 패들렛이 강의 내내 쌓아 온 전시장이자 무대. 꿀팁 3개는 “도구를 다 익히자”가 아니라 “이런 서랍이 있다”를 보여 주는 안내입니다.

🎙️ 무른 경계 원칙

발표가 살아나면 발표에 시간을 더 주세요. 꿀팁 3개는 “다 못 다뤄도 되는 서랍”이라 뒷부분을 유연하게 줄일 수 있습니다. 순서 우선도: 발표 > 꿀팁1(DB) > 꿀팁2(자동화/API) > 꿀팁3(안내).

🎤 40명 발표 운영 — 상세

발표에 쓸 수 있는 시간은 현실적으로 60분 안팎. 40명 전원 발표는 시간상 불가하므로 하이브리드로 갑니다. 강의 내내 써 온 패들렛이 무대입니다.

🎙️ 발표 운영 멘트

“오늘 발표는 패들렛을 미술관처럼 둘러보듯 합니다. 먼저 25분 동안 자유롭게 서로의 앱을 눌러 보고, 좋아요와 한 줄 피드백을 남겨요. 그다음 좋아요 많은 분들이 앞에 나와 2~3분씩 소개합니다. 발표를 안 해도 모두가 이미 발표에 참여한 거예요.”

🎙️ 칭찬의 기준

“완성도가 아니라 어떤 질문을 던졌고 무엇을 발견했는지를 칭찬합니다.” 총평에서도 미완성 앱을 콕 집어 “이 질문이 정말 좋았어요”라고 콕 집어 말해 주면, 못 끝낸 사람도 성공 체험으로 마무리됩니다. 하이라이트 인원 N명은 시계를 보며 유연하게 — 남은 시간 ÷ 3분으로 즉석 계산.

🙋 예상 질문 (발표)

“발표 꼭 무대에서 해야 하나요?” → 아니요. 갤러리 워크로 전원이 이미 발표에 참여합니다. 발표는 자원자만. · “제 앱은 미완성인데요” → 괜찮아요. 질문·데이터·발견 세 줄만 있으면 훌륭한 발표. 미완성도 이야기가 있으면 화면 공유로 발표할 수 있습니다.

블록별 진행 노트

프로젝트 마무리 + 코칭 (0~40′)

🎙️ 멘트 — 세 줄이면 발표 끝

“지금부터 앱을 크게 고치지 마세요. 질문 · 데이터 · 발견 세 줄만 또렷하게 다듬으면 발표 준비 끝이에요. 다 되면 패들렛에 URL을 올려 주세요.”

🧯 예상 오류 (마무리 단계)

URL이 안 열림 → 배포 저장소가 Public인지, Streamlit 앱이 sleep 상태(첫 접속 1~3분)인지 확인. · 보드에 올릴 게 없는 미완자 → 4일차까지 만든 것 중 가장 잘 된 하나를 올리게. 완성 강박을 낮춰 주세요.

[꿀팁 1] 진짜 데이터베이스 (115~150′)

🎙️ 멘트 — 어제의 복습 + 다음 서랍

“어제 Supabase에 데이터를 직접 쌓아 봤죠? 오늘은 그 뒤편을 구경합니다. ‘데이터가 안 보여요’ 하던 잠금장치(RLS)가 왜 있는지, 그리고 데이터가 수만 줄로 커지면 여는 다음 서랍 — SQL로 집계하는 Neon까지. 오늘은 구경만 해요. 지금 다 안 해도 됩니다.”

📖 이론 심화 — RLS(행 수준 보안)

RLS(Row Level Security)는 테이블의 각 행에 대한 접근 규칙. Supabase는 기본이 전부 차단이고, 명시적으로 허용 정책(policy)을 만든 것만 읽기/쓰기가 됩니다. 초보가 “데이터가 안 보여요” 하면 십중팔구 RLS가 켜져 있고 정책이 없는 상태 — 고장이 아니라 안전 기본값. Neon은 순수 PostgreSQL이라 SQL 문장(SELECT … GROUP BY … ORDER BY … LIMIT)으로 집계합니다.

🧯 예상 오류 (DB)

Supabase 데이터가 안 보임(빈 결과) → RLS가 막은 것. 대시보드에서 해당 테이블에 읽기 허용 정책을 추가(또는 실습용이면 RLS 비활성). · 연결 실패 → URL/키 오타, 또는 스트림릿 st.secrets에 연결 문자열 미등록. · secret 키(예전 service_role) 노출 → 마스터 키를 앱/깃허브에 올리면 누구나 전체 접근 가능. 절대 커밋 금지, 반드시 Secrets에만. 실수로 올렸다면 즉시 키 재발급(rotate).

🙋 예상 질문 (DB)

“DB 어려워 보여요” → 지금 안 배워도 됩니다. 필요해질 때(동시 사용·대용량·집계) 배우면 돼요. 그때도 SQL은 AI에게 시키면 됩니다. · “시트랑 뭐가 달라요?” → 시트는 사람이 보는 표, DB는 앱이 빠르게 넣고 꺼내는 창고. 동시 접속·집계에서 차이가 큽니다.

[꿀팁 2] 자동화 + 흥미로운 API (150~175′)

🎙️ 멘트 — 컴퓨터가 대신 모으게

“데이터를 매시간 손으로 모으면 지치죠. GitHub Actions에 시간표(cron)를 적어 두면 컴퓨터가 알아서 모아 줍니다. 그리고 세상엔 공짜로 재밌는 데이터를 주는 API 서랍이 많아요 — 지진, 소행성, 생물, 지도까지.”

📖 이론 심화 — cron 문법

"5 * * * *"는 다섯 칸: 분 시 일 월 요일. 5 * * * *=매시간 5분, 0 3 * * *=매일 3시(UTC). GitHub Actions의 스케줄은 UTC 기준이라 한국시간(KST)은 +9시간. 또한 무료 러너 부하로 정시보다 몇 분 늦게 도는 것이 정상이라, 매시 0분보다 5분처럼 살짝 뒤를 권합니다. workflow_dispatch는 수동 실행 버튼을 추가하는 트리거.

🎙️ 하이라이트 시연 — 예제 저장소 Actions 탭 라이브로 열기

학생 교재의 🛰️ 박스에 맞춰 modudata의 Actions 탭을 화면에 직접 띄우세요. 초록 체크가 줄줄이 쌓인 실행 기록을 보여주며: “여러분이 5일 내내 쓴 기온·인구·지하철 데이터, 사실 이 로봇들이 알아서 갱신하고 있었어요. 데이터가 낡지 않는 비밀이 이겁니다.” 커밋 목록의 🤖 커밋들을 보여주면 각인 효과가 큽니다. 개념(cron) → 실물(Actions 로그) → “여러분 프로젝트에도 붙일 수 있다” 순서로 25분 중 5분 배정.

🧯 예상 오류 (자동화/API)

cron이 한국시간인 줄 알고 9시간 어긋남 → UTC 기준임을 강조(KST = UTC+9). · Actions가 안 도는 것 같음 → 무료 러너는 지연이 흔함. 급하면 workflow_dispatch 수동 버튼으로 즉시 테스트. · data.go.kr 인증키 오류 → serviceKey의 인코딩/디코딩 두 형태 중 맞는 걸로 교체, 응답 포맷은 _type=json. · 키가 코드에 노출 → 반드시 GitHub Secrets/st.secrets로. 카카오는 요청 헤더 Authorization: KakaoAK 형식 지키기.

[꿀팁 3] 다음 걸음 + 마무리 (175~180′)

🎙️ 마무리 멘트 — 리듬을 챙겨 보내기

“오늘 배운 건 도구가 아니라 리듬이에요 — 궁금한 질문 → 데이터 찾기 → AI에게 앱 주문 → 배포 → 의심하며 다듬기. 새 도구는 필요할 때 서랍에서 꺼내면 됩니다. 5일 전 코드가 겁났던 여러분이 지금은 배포하고, 질문하고, 발표합니다. 이미 데이터 분석가예요.” — 박수로 배웅.

🧯 예상 오류 (마무리)

비용 걱정 → 쓴 도구 대부분 무료 한도. 유료 전환은 항상 사전 안내됨. · 키 관리 습관 재강조 → API 키·secret 키(예전 service_role)·비밀번호는 코드에 직접 쓰지 말 것(Secrets/st.secrets). 마지막으로 한 번 더 각인.

✅ 5일차 마무리 체크 (강사)

  • 전원 앱 URL이 패들렛에 게시됐는가
  • 갤러리 워크에서 좋아요·피드백이 오갔는가(전원 참여 확인)
  • 하이라이트 N명 무대 + 강사 총평으로 성공 체험을 남겼는가
  • 꿀팁 3개를 “필요할 때 여는 서랍”으로 안내했는가(강박 없이)
  • 키 관리·비용 안내로 안전하게 배웅했는가
