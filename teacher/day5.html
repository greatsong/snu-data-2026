📖 오늘의 설계 원칙 — 전원 참여 발표

완전 입문 40명. 전원 무대는 시간상 불가하고, 발표 부담도 큽니다. 그래서 갤러리 워크(전원 참여) + 하이라이트 발표(화면 공유)(소수) 하이브리드로 갑니다. 갤러리가 강의 내내 쌓아 온 전시장이자 무대. 심화 내용(DB·자동화·API)은 수강생용 심화 자료 페이지로 분리해, 발표 날의 인지 부하를 줄였습니다. 수업에서는 “이런 자료가 있다”만 보여 주면 됩니다.

“오늘 발표는 갤러리를 미술관처럼 둘러보듯 합니다. 먼저 25분 동안 자유롭게 서로의 앱을 눌러 보고, 좋아요와 한 줄 피드백을 남겨요. 그다음 좋아요 많은 분들이 앞에 나와 2~3분씩 소개합니다. 발표를 안 해도 모두가 이미 발표에 참여한 거예요.”

🎙️ 칭찬의 기준

“완성도가 아니라 어떤 질문을 던졌고 무엇을 발견했는지를 칭찬합니다.” 총평에서도 미완성 앱을 콕 집어 “이 질문이 정말 좋았어요”라고 콕 집어 말해 주면, 못 끝낸 사람도 성공 체험으로 마무리됩니다. 하이라이트 인원 N명은 시계를 보며 유연하게 — 남은 시간 ÷ 3분으로 즉석 계산.

🙋 예상 질문 (발표)

“발표 꼭 무대에서 해야 하나요?” → 아니요. 갤러리 워크로 전원이 이미 발표에 참여합니다. 발표는 자원자만. · “제 앱은 미완성인데요” → 괜찮아요. 질문·데이터·발견 세 줄만 있으면 훌륭한 발표. 미완성도 이야기가 있으면 화면 공유로 발표할 수 있습니다.

블록별 진행 노트

프로젝트 마무리 + 코칭 (0~40′)

🎙️ 멘트 — 세 줄이면 발표 끝

“지금부터 앱을 크게 고치지 마세요. 질문 · 데이터 · 발견 세 줄만 또렷하게 다듬으면 발표 준비 끝이에요. 다 되면 갤러리에 작품을 올려 주세요.”

🎙️ 게시 샘플 먼저 보여주기

수강생용 §1의 게시물 샘플 3개(인구·기온·유튜브)를 화면에 띄우고 “이렇게 올리면 됩니다” 한 번 콕 집어 보여 주세요. 양식은 발표 세 줄과 같습니다 — 제목 = 질문, 본문 = 앱 주소 + 발견 한 줄 + 이름. 샘플을 먼저 보면 “뭘 써야 하죠?” 질문이 거의 사라집니다.

🧯 예상 오류 (마무리 단계)

URL이 안 열림 → 배포 저장소가 Public인지, Streamlit 앱이 sleep 상태(첫 접속 1~3분)인지 확인. · 보드에 올릴 게 없는 미완자 → 4일차까지 만든 것 중 가장 잘 된 하나를 올리게. 완성 강박을 낮춰 주세요.

유연 블록 — 프로젝트 이어가기 + 심화 자료 (115~170′)

🎙️ 운영 — 기본은 프로젝트, 심화는 선택

기본 진행은 발표 피드백 반영 + 순회 코칭입니다. 반 전체가 여유 있으면(또는 희망자가 모이면) 수강생용 심화 자료를 화면에 띄워 아래 순서로 안내하세요. 아래 이론·예상 오류 노트는 심화 자료를 다룰 때만 참고하면 됩니다.

🎙️ 심화 ① 진짜 DB — 4일차 복습에서 출발

“어제 Supabase에 데이터를 직접 쌓아 봤죠? 오늘은 그 뒤편을 구경합니다. ‘데이터가 안 보여요’ 하던 잠금장치(RLS)가 왜 있는지, 그리고 데이터가 수만 줄로 커지면 쓰는 다음 도구 — SQL로 집계하는 Neon까지. 오늘은 구경만 해요. 지금 다 안 해도 됩니다.”

📖 이론 심화 — RLS(행 수준 보안)

RLS(Row Level Security)는 테이블의 각 행에 대한 접근 규칙. Supabase는 기본이 전부 차단이고, 명시적으로 허용 정책(policy)을 만든 것만 읽기/쓰기가 됩니다. 초보가 “데이터가 안 보여요” 하면 십중팔구 RLS가 켜져 있고 정책이 없는 상태 — 고장이 아니라 안전 기본값. Neon은 순수 PostgreSQL이라 SQL 문장(SELECT … GROUP BY … ORDER BY … LIMIT)으로 집계합니다.

🧯 예상 오류 (DB)

Supabase 데이터가 안 보임(빈 결과) → RLS가 막은 것. 대시보드에서 해당 테이블에 읽기 허용 정책을 추가(또는 실습용이면 RLS 비활성). · 연결 실패 → URL/키 오타, 또는 스트림릿 st.secrets에 연결 문자열 미등록. · secret 키(예전 service_role) 노출 → 마스터 키를 앱/깃허브에 올리면 누구나 전체 접근 가능. 절대 커밋 금지, 반드시 Secrets에만. 실수로 올렸다면 즉시 키 재발급(rotate).

🙋 예상 질문 (DB)

“DB 어려워 보여요” → 지금 안 배워도 됩니다. 필요해질 때(동시 사용·대용량·집계) 배우면 돼요. 그때도 SQL은 AI에게 시키면 됩니다. · “시트랑 뭐가 달라요?” → 시트는 사람이 보는 표, DB는 앱이 빠르게 넣고 꺼내는 창고. 동시 접속·집계에서 차이가 큽니다.

🎙️ 심화 ② 자동화 + 흥미로운 API — 깨우기부터 수집까지

도입은 공감 유발로: “이번 주에 만든 앱, 며칠 뒤에 열면 ‘깨우는 중…’ 화면을 만나실 거예요. 무료 앱은 손님이 끊기면 잠들거든요.” → “해결은 간단해요. 로봇에게 시간표를 주고 대신 찾아가게 하면 됩니다. 그 시간표 문법이 cron이에요.” — cron 개념을 깨우기 사례로 먼저 잡아 주고, 그다음 확장합니다: “같은 원리로 시간표에 ‘수집’을 적으면 데이터도 알아서 모입니다. 세상엔 공짜로 재밌는 데이터를 주는 API가 많아요 — 지진, 소행성, 생물, 지도까지.”

📖 이론 심화 — cron 문법

"5 * * * *"는 다섯 칸: 분 시 일 월 요일. 5 * * * *=매시간 5분, 0 3 * * *=매일 3시(UTC). GitHub Actions의 스케줄은 UTC 기준이라 한국시간(KST)은 +9시간. 또한 무료 러너 부하로 정시보다 몇 분 늦게 도는 것이 정상이라, 매시 0분보다 5분처럼 살짝 뒤를 권합니다. workflow_dispatch는 수동 실행 버튼을 추가하는 트리거.

🎙️ 하이라이트 시연 — Actions 탭 두 개를 순서대로

1탄(깨우기): app-caretaker의 Actions 탭을 띄우고 — “오늘 아침에도 이 로봇이 여러분보다 먼저 앱에 다녀갔어요. 매일 07시 23분·19시 23분, 시간표(cron) 두 줄이 전부입니다.” apps.txt를 열어 ‘주소 한 줄 추가가 끝’이라는 것까지 보여주면 자동화의 문턱이 확 낮아집니다. 2탄(수집): 이어서 modudata의 Actions 탭 — “같은 원리로 시간표에 ‘수집’을 적으면? 수업에서 쓴 기온(4일차)·인구(2일차) 데이터가 사실 이 로봇들이 알아서 갱신하고 있었어요.” 커밋 목록의 🤖 커밋들을 보여주면 각인 효과가 큽니다. 공감(내 앱이 잠듦) → 개념(cron) → 실물 2탄(깨우기·수집) 순서로 7분이면 충분합니다. 심화 자료를 못 다루더라도 이 시연만은 마무리 때 보여줄 가치가 있습니다.

🙋 예상 질문 (자동화)

“앱을 몽땅 깨워 두면 안 되나요?” → 무료 자원은 모두가 나눠 씁니다. 지금 쓰는 앱만 돌보미에 올리고, 안 쓰게 된 앱은 apps.txt에서 지우는 게 예의이자 실속입니다. 발표처럼 확실해야 하는 순간엔 직전에 한 번 열어 깨워 두는 것이 가장 확실합니다.

🧯 예상 오류 (자동화/API)

cron이 한국시간인 줄 알고 9시간 어긋남 → UTC 기준임을 강조(KST = UTC+9). · Actions가 안 도는 것 같음 → 무료 러너는 지연이 흔함. 급하면 workflow_dispatch 수동 버튼으로 즉시 테스트. · data.go.kr 인증키 오류 → serviceKey의 인코딩/디코딩 두 형태 중 맞는 걸로 교체, 응답 포맷은 _type=json. · 키가 코드에 노출 → 반드시 GitHub Secrets/st.secrets로. 카카오는 요청 헤더 Authorization: KakaoAK 형식 지키기.

보안이 중요한 데이터 — 로컬 실행 안내 (선택 · 주로 1:1)

🎙️ 누구에게, 언제

“성적·상담 기록·개인정보·병원 자료처럼 남에게 보이면 안 되는 데이터를 다루시는 분 있나요?” 손 드는 분이 있으면(교사·상담·의료·인사 직군에서 자주 나옵니다) 수강생용 §3(로컬 실행)을 함께 엽니다. 전체를 붙잡고 길게 설명하기보다 순회 코칭 중 1:1로 짚는 편이 낫습니다 — 필요한 사람만 정확히 가져가면 됩니다.

🎙️ 핵심 한 문장 — 배포와 로컬의 갈림

“지금까지 한 클라우드 배포는 데이터를 인터넷에 올리는 방식이에요. 민감한 데이터는 그러면 안 되죠. 대신 내 컴퓨터에서만 앱을 켜면(로컬 실행) 데이터가 이 컴퓨터 밖으로 안 나갑니다. 명령은 streamlit run app.py 한 줄, 막히는 곳은 에러를 그대로 복사해 AI에게 물어보면 돼요 — 강의 내내 한 방식 그대로예요.”

📖 이론 심화 — localhost와 Network URL

streamlit run을 실행하면 터미널에 두 주소가 뜹니다. Local URL(localhost:8501)은 그 컴퓨터에서만 열리고, Network URL은 같은 와이파이의 다른 기기에서 접속할 수 있는 주소입니다. 민감 데이터를 다룰 때 강조할 점: Network URL을 공유하면 같은 망의 타인이 데이터를 볼 수 있습니다. localhost만 쓰도록 안내하세요. 로컬 실행이 안전한 이유는 데이터와 API 키(.streamlit/secrets.toml)가 그 컴퓨터에만 있고 어디에도 업로드되지 않기 때문입니다.

🙋 예상 질문 (로컬)

“그럼 남들은 제 앱 못 봐요?” → 네, 그게 목적입니다. 나만 쓰거나 민감 데이터일 때의 방식이에요. 공유가 필요하면 그때 공개해도 되는 데이터로 클라우드 배포. · “파이썬 꼭 깔아야 해요?” → 로컬은 내 컴퓨터가 앱을 돌리므로 파이썬이 필요합니다(클라우드 배포는 스트림릿 서버가 대신 돌려 줬던 것). 설치는 python.org에서 한 번만.

🧯 예상 오류 (로컬 실행)

“streamlit 명령을 못 찾음” → 앞에 python -m(윈도우)·python3 -m(맥)을 붙여 실행. · 윈도우 python이 안 먹힘 → 설치 때 Add Python to PATH 체크 누락 → 그 체크를 켜고 재설치. · “app.py를 못 찾음” → cd로 이동한 폴더에 코드·데이터가 같이 있는지 확인(폴더를 터미널로 드래그하면 경로가 붙음). · 보안 사고 예방 → 민감 데이터 파일을 깃허브에 커밋하거나 Network URL을 공유하지 않도록 마지막에 한 번 더 각인.

마무리 — 심화 자료 안내 + 수료 인사 (170~180′)

🎙️ 마무리 멘트 — 배운 순서를 챙겨 보내기

“심화 자료 페이지에 진짜 DB·자동화·재미있는 API를 정리해 뒀어요. 집에서 꼭 한 번 도전해 보세요. 앞으로도 진행 순서는 같아요 — 궁금한 질문 → 데이터 찾기 → AI에게 앱 주문 → 배포 → 비판적으로 다듬기. 새 도구는 필요할 때 배우면 됩니다. 5일 전 코드가 겁났던 여러분이 지금은 배포하고, 질문하고, 발표합니다. 이미 데이터 분석가예요.” — 박수로 마무리.

🧯 예상 오류 (마무리)

비용 걱정 → 쓴 도구 대부분 무료 한도. 유료 전환은 항상 사전 안내됨. · 키 관리 습관 재강조 → API 키·secret 키(예전 service_role)·비밀번호는 코드에 직접 쓰지 말 것(Secrets/st.secrets). 마지막으로 한 번 더 각인.

✅ 5일차 마무리 체크 (강사)

  • 전원 앱 URL이 갤러리에 게시됐는가
  • 갤러리 워크에서 좋아요·피드백이 오갔는가(전원 참여 확인)
  • 하이라이트 N명 무대 + 강사 총평으로 성공 체험을 남겼는가
  • 심화 자료 페이지를 소개하고 “꼭 도전해 보라”고 안내했는가(강박 없이)
  • 보안 데이터를 다루는 수강생에게 로컬 실행을 안내했는가(해당자만)
  • 키 관리·비용 안내로 안전하게 배웅했는가
