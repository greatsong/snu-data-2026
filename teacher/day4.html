🎙️ 오늘의 목표 선언 (개강 멘트)

“오늘 우리 앱은 기억을 갖게 됩니다 — 점심 메뉴를 투표로 정하고 잔소리까지 하는 앱을 구글 시트에, 약속 잡기 앱을 진짜 DB(Supabase)에 쌓아요. 오후엔 직접 만드는 작은 AI로 2050년 서울 기온을 예측해 봅니다. 코드는 여전히 AI가 짜요.”

📖 오늘의 큰 그림 — 저장 → 학습

오전은 ‘휘발성 극복’(session_state → 구글 시트 → Supabase, 저장소 두 곳을 잇달아), 오후는 ‘남의 AI 쓰기’에서 ‘내 AI 만들기’로의 전환(LLM 호출 → scikit-learn 학습). 두 축이 오늘의 뼈대입니다. ①②③ 세 박자는 오늘도 유지 — 점심 투표 앱·Supabase 앱·예측기 각각 ②(샘플 코드)까지 함께 봅니다.

블록별 진행 노트

복습 (0~15′)

🎙️ 멘트 — 어제의 아쉬움에서 출발

“어제 만든 채팅앱, 멋졌죠? 근데 새로고침하면 대화가 싹 사라져요. 오늘은 그걸 고칩니다.” — 실제로 어제 앱을 새로고침해 데이터가 날아가는 걸 눈앞에서 보여주면 동기부여가 확실합니다.

1. 왜 저장이 필요한가 (15~30′)

📖 이론 심화 — 휘발성과 저장소 선택

session_state는 세션(브라우저 탭) 단위의 임시 메모리. 새로고침·재접속·타 사용자에겐 공유되지 않음. 영속 저장이 필요하면 앱 외부 저장소로. 오늘은 두 저장소를 잇달아 겪는다 — 구글 시트(눈에 보임 · Apps Script 접수창구 방식이라 서비스계정 없음 · 개인 프로젝트 응용에 최적) → Supabase(진짜 DB · 카드 없이 키 두 줄 · 잠금과 동시성). SQL 집계 심화는 심화 자료(Neon).

🙋 예상 질문

“구글 시트도 데이터베이스예요?” → “넓게 보면 ‘데이터를 쌓는 곳’이란 점은 같아요. 다만 시트는 사람이 보는 표에 가깝고, DB는 앱이 쓰는 창고에 가깝죠. 오늘 둘 다 만들어 보면 차이가 몸으로 느껴집니다.”

2. 민주적 점심 — 구글 시트 (30~75′)

🎙️ 멘트 — 잔소리로 문을 여세요

“오늘 첫 앱은 팀 점심을 투표로 정하는 ‘민주적 점심’입니다. 그냥 투표 앱이 아니에요 — 이 앱은 과거를 기억해서, 그저께 먹은 메뉴를 또 고르면 잔소리를 합니다.” 시연 요령: 강사 시트에 미리 ‘김치찌개 · 먹었다’ 기록을 2~3일 전 날짜로 넣어 두고, 수업에서 김치찌개에 투표 → 당선 → “3일 전에도 드셨는데 괜찮으시겠어요? 😏”가 터지는 순간을 보여주세요. ‘저장이 왜 필요한가’가 설명 없이 전달됩니다.

🎙️ 진행 — 관문은 배포·권한 승인 하나

화면 미러링으로 한 계단씩: 시트 생성(머리글 4칸: 시각·팀원·메뉴·구분) → Apps Script 코드 복붙 → 배포(웹 앱 · 실행: 나 · 액세스: 모든 사용자) → 권한 승인(‘확인되지 않은 앱’ 경고 통과) → 중간 점검(주소 뒤 ?member=… 붙여 OK 확인)까지 전원을 통과시킨 뒤에 프롬프트로 넘어갑니다. 중간 점검을 통과한 사람은 이후 무슨 문제가 나든 ‘앱 쪽 문제’로 좁혀지므로 여기서 시간을 아끼지 마세요. 실습은 각자 자기 시트로 — 이름을 바꿔 가며 서너 표를 넣으면 혼자서도 팀 시뮬레이션이 됩니다.

🎙️ 블록 마무리 — 풀버전 30초 시연

실습이 끝나면 강사 화면으로 풀버전(lunch-app · 데모 모드)을 잠깐 보여주세요: 개표식 모드에서 "개표를 시작합니다" → 진행바 → 당선 발표 → 잔소리 말풍선까지. 핵심 멘트: “여러분이 방금 만든 뼈대에 ③ 개선을 거듭하면 여기까지 옵니다. 재료는 똑같이 시트 하나, 주소 하나예요.” 개인 프로젝트 동기부여가 목적이므로 기능 설명은 하지 않습니다.

📖 이론 심화 — Apps Script 웹 앱 = 시트에 딸린 작은 서버

표준 연동(gspread)은 서비스계정 생성 → JSON 키 → 시트 공유 3관문이라 완전입문 최대 난관이었다. Apps Script 웹 앱은 시트 안에서 배포 1관문으로 같은 일을 한다 — 주소로 값이 오면 한 줄 추가(appendRow), 없으면 전체를 JSON으로 반환하는 초소형 API 서버인 셈. 시각을 Utilities.formatDate(…, "Asia/Seoul", …)로 한국 시간 문자열로 박는 이유: Date 객체로 저장하면 JSON으로 나올 때 UTC로 자동 변환돼(9시간 어긋남) 앱 쪽 보정이 필요해지는데, 문자열로 저장하면 이 문제 자체가 사라진다. 앱의 ‘오늘’ 판단도 한국 시간 기준(스트림릿 클라우드 서버는 UTC) — 프롬프트에 이미 못 박혀 있다.

🧯 예상 오류 (빈출)

배포 때 ‘모든 사용자’ 선택지가 없음 → 직장·학교(워크스페이스) 계정. 개인 @gmail.com으로 다시. · 브라우저 실험은 OK인데 앱은 무반응 → 액세스 권한을 ‘나만’으로 배포한 것(조용히 실패). 배포 관리에서 ‘모든 사용자’로 재배포. · SHEET_URL 오답 → /exec로 끝나는 ‘웹 앱 URL’이어야 함(배포 ID·편집 화면 주소 ✗). · ‘확인되지 않은 앱’ 경고에서 멈춤 → 고급 → 이동 → 허용 (정상 경로임을 미리 예고). · Apps Script 수정 후에도 옛날처럼 동작 → 저장만 하고 재배포 안 한 것. 배포 관리 → 새 버전. · 로컬은 되는데 배포하면 키 오류 → 스트림릿 클라우드 Settings → Secrets에 SHEET_URL 미입력. · 투표했는데 개표가 안 뜸 → 시트에 오늘 날짜의 ‘먹고싶다’ 행이 있는지 확인(어제 표는 오늘 개표에 안 들어감).

🙋 예상 질문

“주소만 알면 아무나 내 시트에 쓸 수 있는 거 아니에요?” → “맞아요. 대신 주소가 사실상 비밀번호만큼 길고, 비밀 금고에 넣어 공개도 안 됩니다. ‘진짜 서비스라면 잠금이 필요하다’는 그 감각이 정확해요 — 바로 다음 시간 Supabase의 잠금(RLS)이 그 답입니다.” · “팀원들과 진짜로 같이 쓸 수 있어요?” → “네. 배포한 앱 주소를 팀 채팅방에 올리면 끝. 오늘은 각자 자기 시트로 연습하고, 진짜 팀 운영은 응용 과제로.” · “시트를 직접 고쳐도 돼요?” → “돼요! 그게 시트 저장소의 매력이에요. 오타 난 줄을 지우거나 고치면 앱에도 바로 반영됩니다.”

3. Supabase 연동 앱 (85~115′)

🎙️ 멘트 — 카드도 서비스계정도 없이

“오전엔 시트, 이번엔 진짜 데이터베이스예요. 원리는 아까와 똑같아요(앱↔저장소). Supabase는 카드 등록도, 복잡한 로봇 계정도 없이 URL·키 두 줄이면 끝나요.” 가입은 이 시간에 다 함께 합니다(숙제 아님) — 화면 미러링으로 한 계단씩: 가입(처음이면 조직 만들기가 먼저 나옴 · 아무 이름이나 · 이미 계정 있으면 로그인만) → New project(생성 대기 1~2분은 다음 설명을 들으며) → 수강생 교재의 SQL 복붙 한 번으로 votes 테이블(name·times) 생성 + RLS 끄기를 동시에 → ✔️ 30초 확인(빨간 RLS disabled 배지가 보이는지)을 전원 통과시키고 → Settings→API Keys에서 publishable 키, Settings→Data API에서 API URL 복사(.supabase.co까지만, 뒤 /rest/v1 있으면 지우기). 주의: Insert row로 한 줄 넣어 보는 방식은 확인이 안 됩니다 — 대시보드는 관리자 권한이라 RLS를 무시하고 통과하므로, 반드시 배지로 확인시키세요.

📖 딱 하나의 관문 — RLS

Supabase 새 테이블은 RLS(Row Level Security)가 기본 ON이라 정책이 없으면 막힙니다("데이터가 안 보여요"의 99%). 증상은 두 갈래 — 저장(insert)은 row-level security 문구가 든 빨간 에러가 크게 나고, 읽기(select)는 에러 없이 빈 목록이라 조용히 지나갑니다. 연습용은 두 길: ① 테이블 설정에서 RLS 비활성화(가장 간단), ② Authentication→Policies에서 모두 허용 정책 추가. 이 한 가지만 미리 못 박으면 전원 통과. (심화 자료(extra.html)에서 RLS를 '왜 기본으로 잠그나'로 다시 다룸.)

🧯 예상 오류 (빈출)

제출 시 row-level security 빨간 에러 / 에러 없이 명단만 계속 빈 채 → 둘 다 1순위 RLS. 테이블 잠금을 풀었는지 확인. · 키 혼동 → 앱에는 publishable 키(예전 anon), secret 키(예전 service_role)는 절대 앱에 넣지 말 것. · 테이블/열 이름 불일치 → 프롬프트의 votes·name·times와 실제 테이블이 같은지. · 모듈 없음 → requirements.txt에 supabase 추가. · 새 프로젝트 생성이 거부될 때 → 무료 플랜은 활성 프로젝트 2개까지, 기존 계정 소유자는 안 쓰는 프로젝트를 일시정지(pause) 후 재시도. · 프로젝트 잠들었을 때(7일 미사용) → 대시보드에서 재개(Resume). 강의 중엔 거의 발생 안 함.

🙋 예상 질문

“돈 나가요? 카드 필요해요?” → 둘 다 아니요. 무료 플랜은 카드 없이 영구 무료(DB 500MB·활성 프로젝트 2개 — 이미 2개라면 안 쓰는 것을 일시정지하면 됨). · “아까 시트로 됐는데 왜 또 DB예요?” → “시트는 눈에 보이고 손으로 고치기 좋지만, 사람이 많아지고 데이터가 커지면 잠금·검색·동시 접속이 필요해요. 그때의 답이 DB — 둘 다 겪어 보면 프로젝트에서 골라 쓸 수 있어요.” · “RLS가 뭐예요?” → “DB의 기본 잠금장치. 켜져 있어 안 보이는 거라 고장 아님 — 연습용은 잠깐 꺼둡니다.”

4. 머신러닝 맛보기 (115~160′)

🎙️ 멘트 — “직접 만드는 작은 AI”

“지금까진 남이 만든 AI를 불러다 썼죠. 이번엔 여러분이 데이터로 직접 아주 작은 AI를 학습시켜요.” 목표는 완벽한 예측이 아니라 ‘학습이란 이런 것’이라는 체험. model.fit() 한 줄에서 “지금 AI가 태어났어요!”로 감탄하게 만듭니다. 진행 요령: 4-1·4-2 개념 표는 앞에서 길게 강의하지 말고 5~7분으로 훑고 바로 실습으로 — 수강생 교재에 "지금 외우지 않아도 됨"과 "방금 코드가 5단계 그대로" 박스가 있으니, 코드가 돌아간 뒤 그 박스로 다시 확인하는 쪽이 완전 입문에 훨씬 잘 남습니다. 오늘은 저장 실습이 둘이라 시간이 빠듯하니 4-4 영화 예측기는 숙제·자습으로 안내하세요(교재에 전체가 있음).

📖 이론 심화 — 회귀와 R²

선형회귀는 데이터에 가장 잘 맞는 직선(오차 제곱합 최소)을 찾는 것. R²(결정계수)는 0~1, 모델이 데이터의 변동을 얼마나 설명하는지의 비율 — 1에 가까울수록 설명력이 큼. 단, R²가 높다고 미래 예측이 정확하단 뜻은 아님(훈련 범위 안에서의 적합도일 뿐).

📖 이론 심화 — 과적합 · 외삽 · 상관≠인과

과적합(overfitting): 고차 다항식으로 훈련 점을 과하게 맞추면 훈련 데이터엔 완벽하지만 새 데이터엔 엉망 → 그래서 오늘은 직선(1차)으로 단순하게. · 외삽(extrapolation): 훈련 범위 밖 예측은 위험 — 2050년은 데이터에 없으므로 직선을 늘인 ‘가정’일 뿐. 수강생에게 반드시 ‘조심 딱지’로 각인. · 상관≠인과: 연도와 기온이 함께 오른다고 “연도가 기온을 올린다”는 아님(뒤에 도시화·온실가스 등 실제 원인). 예측기는 패턴 요약이지 원인 규명이 아님을 강조.

📖 이론 심화 — 전쟁 결측 4년, 지워도 되는 이유 (“시계열인데 잘라도 돼요?”)

회귀는 시간을 한 걸음씩 걷는 방법이 아니라, 114년치 점을 한 화면에 뿌려 놓고 자를 눕히는 방법 — 필요한 건 점의 위치지 연속이 아님. 비유: “114장짜리 사진첩에서 4장 빠진 것이지, 필름 끊긴 영화가 아니에요.” · 직선의 기울기는 시소처럼 축(가운데)에서 먼 양 끝 점이 결정 — 1950~53년은 정확히 ‘축 옆 구경꾼’ 자리. 실측: 가운데 20년을 지워도 2050년 예측 변화 0.02℃, 최근 10년을 지우면 0.26℃(끝이 10배 이상 소중). · 단, 이동평균·전년 대비 변화율처럼 ‘어제 값으로 오늘을 계산’하는 방법은 시간을 걸어가므로 중간 구멍에 실제로 빠짐 — 우리 실습(연도를 좌표로 쓰는 회귀)은 해당 없음. · 라이브 데모 소재: 전처리 필터를 빼고 돌리면 1953년 연평균이 0.6℃(겨울 31일만)로 찍혀 R²가 0.70→0.39로 붕괴 — 결측 처리 규칙이 왜 필요한지 보여주는 실물 반례.

📖 이론 심화 — “최근 30년만으로 예측하면 안 돼요?” (창 길이의 통계학)

학습 창을 줄여 같은 직선을 그어 보면(실측): 전체 114년 +2.60±0.16 · 50년 +3.96±0.56 · 30년 +3.82±1.19 · 20년 +8.14±1.99 ℃/100년 — 창이 짧을수록 기울기는 가팔라지고 불확실성이 폭발(30년에서 이미 7배). 2045년 예측도 13.9℃→15.7℃로 갈라짐. · 짧은 창의 장점 = 최근 가속을 반영(편향↓), 대가 = 더운 해 몇 개에 직선이 휘둘림(분산↑) — 교과서적 편향·분산 맞바꿈. · 이 갈라짐이 통계적 우연(노이즈)인지 진짜 최근 변화(가속)인지는 점 추정치 하나로 절대 구분 못 하고, 오차 막대(신뢰구간)와 정식 검정이 있어야 구분됨. 실측: 50년 창의 95% 구간(+2.85~+5.07)이 전체 기울기 +2.60을 벗어나고, 가속(2차항) 검정도 t=3.46으로 유의 → 서울의 온난화 가속은 실제. 반면 30년 창 구간(±2.35)은 +2.60을 포함 — 30년치만으론 가속을 단정할 수 없음. 20년 이하는 구간이 지나치게 넓고 독립성 가정까지 흔들려 판단 재료로 부적합. · 정리: ‘어느 창으로 긋나’라는 선택의 차이(1.8℃)가 통계 오차(±1.2~1.6℃)와 맞먹는다 — 진짜 기후 예측을 직선이 아니라 물리 기반 시나리오 모델로 하는 이유. · 수업 활용: 여유가 되면 “전체로 한 번, 최근 30년으로 한 번 그어 보세요” → 예측이 갈라지는 걸 직접 보는 체험이 ‘조심 딱지’(외삽 경고)의 최고 교보재.

📖 이론 심화 — 백테스트: “그때 예측했다면 맞았을까?” (방법의 정직한 검증)

미래 예측이 통계적으로 의미 있는지 확인하는 가장 정직한 방법 = 백테스트: 과거의 한 시점까지만 학습시키고 그 뒤를 예측해 실제와 대조. “2005년에 서서 2016~25년 평균(실제 13.67℃)을 예측했다면”의 실측:

방법 (2005년까지만 학습)예측오차
전체 직선 (1908~2005)13.05℃−0.62℃
최근 50년 직선13.50℃−0.17℃
2차곡선 (가속 반영)13.72℃+0.05℃

10~20년 지평에선 가속 반영 모델이 오차 0.2℃ 이내(자연 변동 ±0.6℃보다 작음)로 실제 작동. 반면 같은 실험을 1985년에서 돌리면(30~40년 뒤 예측) 직선·2차곡선 모두 약 1℃씩 밑나감 — 그 뒤에 일어난 가속을 어떤 과거 데이터도 담지 못했기 때문. 지금의 2050년 예측도 모델별로 14.0~15.0℃로 1℃가 갈라지며, 이 간극은 데이터를 더 모아도 줄지 않음. · 정리: 10~20년 = ‘가속 반영 모델 + 예측구간 + 백테스트 검증’으로 통계적으로 의미 있는 예측 가능 · 25년 이상 = 통계가 아니라 시나리오의 영역(기후 기관들이 회귀가 아닌 물리 모델 + 배출 시나리오를 쓰는 이유). ‘조심 딱지’가 가리키는 경계가 정확히 여기.

🧯 예상 오류

ModuleNotFoundError → requirements.txt에 scikit-learn(설치명, import는 sklearn) 확인. · X 모양 오류 → yearly[["연도"]] 대괄호 두 개(2차원)여야 함, yearly["연도"](1차원)는 에러. AI가 준 코드면 대개 정상. · CSV 열 이름 불일치 → 실제 열은 날짜·지점·평균기온·최저기온·최고기온(한국어). 영어 열 이름(year·temp)을 가정한 코드면 프롬프트의 열 안내를 다시 붙여넣기. · 전처리 누락(가장 흔함) → 1951~52년은 통째 결측이라 그대로 학습하면 NaN 에러, 1953년은 31일 관측뿐이라 연평균 0.6℃ 이상치(R²가 0.70→0.39로 추락). "진행 중인 올해와 관측일 부족한 해는 빼줘"가 프롬프트에 있는지 확인. · predict 경고 → model.predict([[year]]) 형태 유지.
