데이터 저장과 머신러닝 — 강사 진행안

학생용 4일차 페이지 ↗ 를 화면에 띄운 채 진행합니다. 여기에는 시간표 · 멘트 · 예상 질문 · 예상 오류 · 이론 심화를 덧댔습니다.

🎙️ 오늘의 목표 선언 (개강 멘트)

“오늘 우리 앱은 기억을 갖게 됩니다 — 데이터를 진짜 DB(Supabase)에 쌓아요. 그리고 오후엔 직접 만드는 작은 AI로 2050년 서울 기온을 예측해 봅니다. 코드는 여전히 AI가 짜요.”

📖 오늘의 큰 그림 — 저장 → 학습

오전은 ‘휘발성 극복’(session_state → 외부 저장소=Supabase), 오후는 ‘남의 AI 쓰기’에서 ‘내 AI 만들기’로의 전환(LLM 호출 → scikit-learn 학습). 두 축이 오늘의 뼈대입니다. ①②③ 세 박자는 오늘도 유지 — Supabase 앱과 예측기 각각 ②(샘플 코드)까지 함께 봅니다.

🎙️ 무른 경계 원칙

Supabase는 카드·서비스계정 없이 키 두 줄이라 옛 구글시트보다 훨씬 빨라, 최대 난관이 사라졌습니다. 그래도 막히는 곳은 딱 하나 — RLS(잠금) 탓에 데이터가 안 보이는 것. "새 테이블은 기본 잠김 → 연습용은 RLS 끄기/모두 허용 정책"을 미리 못 박으면 대부분 통과. 밀리면 ML을 먼저 짧게 시연해 성취감 확보.

블록별 진행 노트

복습 (0~20′)

🎙️ 멘트 — 어제의 아쉬움에서 출발

“어제 만든 채팅앱, 멋졌죠? 근데 새로고침하면 대화가 싹 사라져요. 오늘은 그걸 고칩니다.” — 실제로 어제 앱을 새로고침해 데이터가 날아가는 걸 눈앞에서 보여주면 동기부여가 확실합니다.

1. 왜 저장이 필요한가 (20~40′)

📖 이론 심화 — 휘발성과 저장소 선택

session_state는 세션(브라우저 탭) 단위의 임시 메모리. 새로고침·재접속·타 사용자에겐 공유되지 않음. 영속 저장이 필요하면 앱 외부 저장소로. 오늘 Supabase를 고르는 이유: 진짜 DB인데 카드·서비스계정 없이 키 두 줄이라 완전입문에 가장 마찰이 적다. 눈에 보이는 구글시트는 매력적이나 서비스계정 세팅이 무거워 데모로. SQL 집계 심화는 5일차(Neon).

🙋 예상 질문

“왜 구글시트 안 쓰고 Supabase예요?” → “구글시트는 서비스계정·JSON키·시트공유 3단 세팅이 초보에게 최대 난관이라, 오늘은 키 두 줄이면 끝나는 Supabase로 진짜 DB를 바로 경험합니다. 구글시트는 원리만 화면으로 보여드려요.”

2. Supabase 연동 앱 (40~80′)

🎙️ 멘트 — 카드도 서비스계정도 없이

“진짜 데이터베이스에 저장할 거예요. Supabase는 카드 등록도, 복잡한 로봇 계정도 없이 URL·키 두 줄이면 끝나요.” 화면 미러링으로 함께: 가입 → New project → Table Editor로 votes 테이블(name·times) → RLS 끄기 → Settings→API Keys에서 publishable 키, Settings→Data API에서 Project URL 복사. 옛 구글시트 방식(서비스계정·JSON·공유)은 맨 끝에 화면으로만 잠깐 보여주고 넘어갑니다.

📖 딱 하나의 관문 — RLS

Supabase 새 테이블은 RLS(Row Level Security)가 기본 ON이라 정책이 없으면 insert/select가 조용히 막힙니다("데이터가 안 보여요"의 99%). 연습용은 두 길: ① 테이블 설정에서 RLS 비활성화(가장 간단), ② Authentication→Policies에서 모두 허용 정책 추가. 이 한 가지만 미리 못 박으면 전원 통과. (5일차 꿀팁에서 RLS를 '왜 기본으로 잠그나'로 다시 다룸.)

🧯 예상 오류 (빈출)

데이터가 안 쌓임/안 보임 → 1순위 RLS. 테이블 잠금을 풀었는지 확인. · 키 혼동 → 앱에는 publishable 키(예전 anon), secret 키(예전 service_role)는 절대 앱에 넣지 말 것. · 테이블/열 이름 불일치 → 프롬프트의 votes·name·times와 실제 테이블이 같은지. · 모듈 없음 → requirements.txt에 supabase 추가. · 프로젝트 잠들었을 때(7일 미사용) → 대시보드에서 재개(Resume). 강의 중엔 거의 발생 안 함.

🙋 예상 질문

“돈 나가요? 카드 필요해요?” → 둘 다 아니요. 무료 플랜은 카드 없이 영구 무료(DB 500MB·프로젝트 2개). · “구글시트로 하면 안 돼요?” → “돼요, 다만 서비스계정·JSON·시트공유 세팅이 훨씬 무거워서 오늘은 강사 데모로만. 원리는 똑같아요(앱↔저장소).” · “RLS가 뭐예요?” → “DB의 기본 잠금장치. 켜져 있어 안 보이는 거라 고장 아님 — 연습용은 잠깐 꺼둡니다.”

3. 머신러닝 맛보기 (90~155′)

🎙️ 멘트 — “직접 만드는 작은 AI”

“지금까진 남이 만든 AI를 불러다 썼죠. 이번엔 여러분이 데이터로 직접 아주 작은 AI를 학습시켜요.” 목표는 완벽한 예측이 아니라 ‘학습이란 이런 것’이라는 체험. model.fit() 한 줄에서 “지금 AI가 태어났어요!”로 감탄하게 만듭니다.

📖 이론 심화 — 회귀와 R²

선형회귀는 데이터에 가장 잘 맞는 직선(오차 제곱합 최소)을 찾는 것. R²(결정계수)는 0~1, 모델이 데이터의 변동을 얼마나 설명하는지의 비율 — 1에 가까울수록 설명력이 큼. 단, R²가 높다고 미래 예측이 정확하단 뜻은 아님(훈련 범위 안에서의 적합도일 뿐).

📖 이론 심화 — 과적합 · 외삽 · 상관≠인과

과적합(overfitting): 고차 다항식으로 훈련 점을 과하게 맞추면 훈련 데이터엔 완벽하지만 새 데이터엔 엉망 → 그래서 오늘은 직선(1차)으로 단순하게. · 외삽(extrapolation): 훈련 범위 밖 예측은 위험 — 2050년은 데이터에 없으므로 직선을 늘인 ‘가정’일 뿐. 학생에게 반드시 ‘조심 딱지’로 각인. · 상관≠인과: 연도와 기온이 함께 오른다고 “연도가 기온을 올린다”는 아님(뒤에 도시화·온실가스 등 실제 원인). 예측기는 패턴 요약이지 원인 규명이 아님을 강조.

🧯 예상 오류

ModuleNotFoundError → requirements.txt에 scikit-learn(설치명, import는 sklearn) 확인. · X 모양 오류 → yearly[["연도"]] 대괄호 두 개(2차원)여야 함, yearly["연도"](1차원)는 에러. AI가 준 코드면 대개 정상. · CSV 열 이름 불일치 → 실제 열은 날짜·지점·평균기온·최저기온·최고기온(한국어). 영어 열 이름(year·temp)을 가정한 코드면 프롬프트의 열 안내를 다시 붙여넣기. · 전처리 누락(가장 흔함) → 1951~52년은 통째 결측이라 그대로 학습하면 NaN 에러, 1953년은 31일 관측뿐이라 연평균 0.6℃ 이상치(R²가 0.70→0.38로 추락). "진행 중인 올해와 관측일 부족한 해는 빼줘"가 프롬프트에 있는지 확인. · predict 경고 → model.predict([[year]]) 형태 유지.

🙋 예상 질문

“예측을 믿어도 돼요?” → “훈련 범위 안은 참고할 만해요. 하지만 2050년 같은 미래는 외삽이라 조심해야 해요. 직선을 늘인 가정일 뿐, 실제 미래는 다르게 갈 수 있어요.” · “군집(KMeans)은 뭐예요?” → “정답 없이 비슷한 것끼리 자동으로 묶는 비지도학습이에요. 오늘은 개념만, 필요하면 5일차 프로젝트에서.”
