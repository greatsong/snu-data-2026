코드가 겁나지 않아요 — 강사 진행안

수강생용 1일차 페이지 ↗ 를 화면에 띄운 채 진행합니다. 여기에는 시간표 · 멘트 · 예상 질문 · 예상 오류 · 이론 심화를 덧댔습니다.

🎙️ 오늘의 목표 선언 (개강 멘트)

“오늘 우리는 코드를 짜지 않습니다. 읽고, AI에게 시키고, 내 인터넷 주소를 3개 만들어 집에 갑니다. 하나라도 성공하면 오늘은 대성공이에요.”

📖 교재의 핵심 장치 — ①②③ 세 박자

수강생 교재의 모든 만들기는 ① 샘플 프롬프트(청록) → ② AI가 준 코드 예시(갈색) → ③ 프롬프트 개선 예시(주황) 순서로 배치돼 있습니다. 이게 바이브 코딩의 리듬이에요. 첫 활동(MBTI)에서 세 블록을 천천히 설명하며 가르치고, 이후엔 “①·③ 기억나죠?”로 빠르게 넘기면 됩니다. ②(샘플 코드)는 “여러분 결과가 이 모습이면 성공”이라는 안심용 — 똑같지 않아도 된다고 강조.

🎙️ 무른 경계 원칙

블록 3(주식앱)이 밀리면 2일차 워밍업으로 이월. 완전 입문 40명에서 ‘전원 성공’이 ‘진도 완주’보다 중요합니다. 조별로 속도 편차가 크면 빠른 조에게는 “색·카드 더 꾸며보기”를 던져 대기시간을 메웁니다.

블록별 진행 노트

OT + 바이브코딩 개념 (0~25′)

🎙️ 멘트 — 요리사 비유

“코드를 짜는 건 AI라는 요리사, 우리는 주문하는 주인이에요. 다만 이 요리사가 가끔 소금을 설탕으로 착각해요. 그래서 우리는 요리 과정을 곁눈으로 읽을 줄만 알면 됩니다.”

📖 이론 심화 — ‘바이브 코딩’

2025년 안드레이 카파시가 쓴 표현. 사람이 자연어로 의도를 말하고 AI가 코드를 생성, 사람은 문제 정의·검증·반복을 맡는 방식. 이 강의 전체의 뼈대. 수강생 교재에는 용어만, 배경은 여기서.

0. 코드 읽는 눈 (25~50′)

🎙️ 멘트

“지금부터 딱 다섯 개만 볼게요. 외우지 마세요. 앞으로 코드에서 계속 만날 거예요. 그때 ‘아, 그거’ 하면 됩니다.” — 다섯 생김새를 df.head() 로 합쳐 읽어 보이기.

🙋 예상 질문

“왜 리스트는 0부터 세나요?” → “맨 앞에서 0칸 떨어진 게 첫 번째”라는 컴퓨터의 오랜 약속. 처음엔 다 헷갈림, 몇 번 보면 익음. · “=랑 ==는 뭐가 달라요?” → =는 담아라, ==는 같니? 라고 묻기.

🧯 예상 오류 — 개념 방어

여기서 실습 에러는 없지만, “외워야 하나”라는 불안이 가장 큰 장애물. “코드를 짠 건 AI, 외우는 사람은 없다”를 반복해 심리적 문턱을 낮춥니다.

1. GitHub + 첫 배포 — MBTI 테스트 (60~105′)

🎙️ 멘트 — 재미로 시작, 첫 성공 크게 축하

“첫 배포는 재미있게 — MBTI 테스트를 AI에게 만들게 해서 인터넷에 올릴 거예요.” 주소가 뜨는 순간 “방금 인터넷에 사이트를 배포했어요! 이미 개발자예요.” — 박수. 이 감정이 5일을 끌고 갑니다.

📖 왜 첫 배포에 AI 생성을 쓰나 (안전한 이유)

MBTI는 외부 데이터·인터넷 연결이 필요 없어 AI가 만든 HTML이 파일 하나로 자기완결됩니다(날씨 대시보드보다 실패 위험 낮음). 그래서 ‘GitHub 처음’ + ‘AI 생성 처음’을 함께 태워도 안전. 프롬프트에 “파일 하나·외부 연결 없이”를 꼭 넣게 하세요.

🧯 예상 오류 (빈출)

Pages 주소가 404 → ①파일명이 정확히 index.html인지 ②Settings→Pages에서 Branch가 main인지 ③1~2분 기다렸는지. · 한글 깨짐 → <meta charset="utf-8"> 있는지(AI에 “utf-8 넣어줘”). · 버튼이 안 눌림/결과 안 뜸 → AI가 JS를 파일 밖에 두려 한 경우. “CSS와 자바스크립트를 index.html 안에 넣어줘”로 재요청. · Settings에 Pages가 안 보임 → 저장소가 Private → Public으로.

🙋 예상 질문

“커밋이 뭐예요?” → ‘이 상태로 저장’ + ‘저장 기록 남기기’. 나중에 되돌리기도 됨(History). 지금은 ‘저장 버튼’으로만 이해해도 충분.

2. HTML 날씨 대시보드 (105~140′)

🎙️ 멘트 — 대화로 고치기 시범

받은 결과를 띄우고 일부러 한 번 고쳐 보이기: “카드를 4개로 늘려줘”, “색을 주황으로”. “보세요, 대화로 고치는 것이 바이브 코딩이에요.”

🧯 예상 오류

데이터 URL 오타/한글깨짐 → 프롬프트의 raw URL을 그대로 복붙하게. · 그래프가 안 뜸 → AI가 CDN 라이브러리를 빠뜨린 경우. “그래프 라이브러리를 CDN으로 넣어줘”라고 이어 말하기. · 파일명을 weather.html로 안 만듦 → index.html을 덮어써 MBTI 페이지가 사라짐 → 새 파일로.

📖 이론 심화 — 정적 vs 동적

HTML은 만든 순간의 숫자로 고정(정적). 사용자가 값을 바꿔 다시 계산하게 하려면 서버에서 코드가 실행(동적)돼야 함 → 다음 블록 스트림릿의 존재 이유. 이 대비를 여기서 심어두면 3번이 자연스럽게 열립니다.

3. 스트림릿 왜? + 주식앱 (150~180′)

🎙️ 멘트 — 전환 훅

“방금 날씨 대시보드, 멋지죠? 근데 숫자가 고정이에요. 사용자가 ‘난 카카오를 볼래’ 하고 직접 바꾸게 하려면? — 그래서 스트림릿을 배웁니다.”

🧯 예상 오류 (배포 빈출)

ModuleNotFoundError → requirements.txt 누락/오타. streamlit·yfinance·plotly 세 줄 확인. · 배포 시 저장소가 안 보임 → Streamlit이 GitHub 연동 안 됨(준비②로). · yfinance 빈 그래프 → 종목 코드 형식(한국주식 .KS 접미사). 예시 005930.KS. · 앱이 계속 running → 첫 배포는 1~3분 걸림, 기다리기.

🙋 예상 질문

“내 앱을 남이 봐도 되나요?” → 공개 앱이라 URL 있으면 접속 가능. 개인정보·비밀키는 안 넣으면 안전(3일차에 키 보관법 배움). · “왜 이렇게 느려요?” → 무료 클라우드라 처음 깨어날 때 느림. 정상.

🎙️ 마무리 멘트 + 씨앗

“오늘 만든 주소 3개를 꼭 메모하세요. 마지막 날 여러분 프로젝트의 재료가 됩니다. 내일은 이 도구로 진짜 데이터를 다뤄요.”
