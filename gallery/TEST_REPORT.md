## 2. 발견 결함 (심각도순)

### [중간 심각도] likes 컬럼을 INSERT 시점에 임의 값으로 조작 가능

- **재현 절차**:
  ```bash
  curl -X POST "https://ipcherzsnaevkkjrclvn.supabase.co/rest/v1/apps" \
    -H "apikey: <publishable key>" -H "Authorization: Bearer <publishable key>" \
    -H "Content-Type: application/json" \
    -d '{"assignment":"1일차 MBTI 페이지","nickname":"아무닉네임","url":"https://example.com/","description":"설명","likes":9999}'
  ```
  → HTTP 201로 성공, `likes: 9999`인 새 행이 생성됨. 실제 갤러리 화면에서도 🩷 9999로 노출됨(테스트 행 "테스트-likes조작"으로 확인 가능).
- **원인**: `apps` 테이블의 insert RLS 정책이 `with check (true)`로 전체 허용이라, 클라이언트가 보내는 JSON 바디의 모든 컬럼(위조된 `likes` 포함)이 그대로 저장됨. `increment_likes` RPC는 **기존 행의 UPDATE**만 막아주고, **신규 행 INSERT 시 초기 likes 값 위조**는 막지 못함.
- **영향**: 누구나 REST API를 직접 호출하는 방법을 알면(공개 publishable 키만 있으면 됨) 원하는 좋아요 수를 가진 가짜 게시물을 만들 수 있음. 기존 게시물의 좋아요를 변경할 수는 없으므로 영향 범위는 제한적이나, 갤러리의 "좋아요 순위"에 대한 신뢰도를 훼손할 수 있음.
- **참고 수정 방향(적용은 하지 않음, 보고만)**: `apps` 테이블에 `likes` 컬럼을 insert 시 항상 0으로 강제하는 `BEFORE INSERT` 트리거를 추가하거나, insert 정책에서 `with check (true)` 대신 `likes = 0`을 조건으로 추가.

### [경미] 모바일(375px) 뷰에서 상단 헤더 텍스트 줄바꿈 깨짐

- **재현 절차**: 브라우저 뷰포트를 375×812(모바일)로 설정 후 갤러리 페이지 로드.
- **증상**: `.topbar .inner`가 `display:flex`이고 `.brand`에 `white-space:nowrap`이 없어, 좁은 화면에서 "병아리반 작품 갤러리" 텍스트가 한 글자씩 세로로 줄바꿈되어 매우 부자연스럽게 표시됨(스크린샷 확인, 파일: `style.css` 49~69번째 줄 `.topbar`/`.brand` 관련 규칙).
- **영향**: 시각적 문제만 있고 기능(제출·필터·좋아요·피드백)에는 영향 없음. 실제 수강생 다수가 모바일로 접속할 가능성이 있어 첫인상에 영향.
- **참고 수정 방향(적용은 하지 않음)**: `@media (max-width: 640px)` 블록에 `.brand { font-size: 0.95rem; }` 또는 `.topbar .inner { flex-wrap: wrap; }` 등 추가.

## 3. 참고사항 (결함은 아니지만 기록)

1. **RLS 거부 시 HTTP 상태코드**: update/delete 시도가 거부될 때 PostgREST는 403이 아니라 200(빈 배열)/204(No Content)를 반환합니다. 이는 "권한 정책이 아예 없으면 대상 행이 0건으로 필터링된다"는 PostgreSQL RLS의 정상 동작이며, 실제 데이터는 안전하게 보호되고 있음을 재조회로 확인했습니다. 다만 응답 코드만 보고 "성공"이라 오인하지 않도록 유의가 필요합니다.
2. **좋아요 재클릭 방지는 브라우저(localStorage) 단위**: 같은 브라우저에서는 재클릭이 막히지만, 시크릿 모드나 다른 브라우저/기기에서는 같은 게시물에 다시 좋아요를 누를 수 있습니다. 로그인 없는 앱 특성상 설계 의도로 보이며 결함으로 분류하지 않았습니다.
3. **XSS 방어는 튼튼함**: `app.js`가 사용자 데이터를 표시할 때 전부 `textContent`를 사용(`innerHTML` 미사용)하고 있어, DB에 원문 그대로 저장된 위험한 문자열(`<script>`, `<img onerror>` 등)도 브라우저에서는 항상 안전한 텍스트로만 렌더링됩니다.

## 5. 교사 확인 가이드 (육안 확인용, 5가지 포인트)

실제 배포 갤러리(https://greatsong.github.io/snu-data-2026/gallery/)에서 다음을 확인하면 이번 테스트가 정상적으로 실행되었음을 알 수 있습니다.

1. **전체 카드 수 확인**: 갤러리 상단 "총 16개 작품"이 보이고, 카드 닉네임이 모두 "테스트"로 시작하면 정상입니다(= 이번 테스트로 생성된 데이터입니다. 실제 수강생 데이터가 섞이면 그만큼 개수가 더 많아집니다).
2. **필터 동작 확인**: 상단 필터 칩("1일차 · MBTI 페이지" 등)을 하나씩 눌러보세요. 클릭할 때마다 해당 산출물만 남고 개수가 즉시 바뀌면 정상입니다.
3. **XSS 안전성 확인(중요)**: "1일차 · MBTI 페이지" 필터를 누르면 닉네임이 "테스트XSS-desc", "테스트XSS-img", "테스트XSS<svg onload=x>"인 카드가 보입니다. 이 카드들을 열었을 때 **경고창(alert)이 뜨지 않고**, 설명란에 `<script>alert(1)</script>` 같은 문자가 **글자 그대로** 보이면 안전하게 처리된 것입니다.
4. **결함 재현 카드 확인**: 같은 필터에서 "테스트-likes조작" 카드의 좋아요 수가 🩷 9999로 표시됩니다. 이는 정상적인 사용자 클릭이 아니라 이번 테스트에서 REST API로 직접 조작해 만든 결함 재현용 데이터입니다(위 "발견 결함" 참고).
5. **정리는 교사 몫**: 확인이 끝나면 `cleanup_test.sql`을 Supabase SQL Editor에서 실행해 "테스트"로 시작하는 행을 한 번에 삭제할 수 있습니다. 삭제 전까지는 지우지 않고 그대로 남겨두었습니다.
