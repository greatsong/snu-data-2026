심화 서랍 — 더 가고 싶을 때

수업에서 다 다루지 않는 다음 걸음 3가지를 모아 둔 서랍입니다. 지금 다 읽지 않아도 됩니다. 필요해질 때(데이터가 커질 때, 매일 손으로 모으기 지칠 때, 새 재료가 궁금할 때) 하나씩 꺼내 쓰세요.

💡 서랍 여는 순서 (추천)
  • 프로젝트를 계속 키우다가 막히는 순간이 오면 그때 해당 서랍부터
  • ① 진짜 DB — 여러 사람이 쓰거나 데이터가 커질 때
  • ② 자동화 + API — 매일 손으로 모으기 지칠 때, 새 데이터가 궁금할 때
  • ③ 다음 걸음 — 새 프로젝트 재료를 찾을 때

1. 진짜 데이터베이스 — 4일차 Supabase, 한 겹 더

  • 4일차에 = Supabase(DB)에 약속/투표 데이터를 직접 쌓아 봄
  • 여기서는 = 그 잠금장치(RLS)의 이유 복습 + SQL 집계(Neon) 구경
🤔 언제 구글시트 대신 DB?
  • ① 여러 사람 동시 저장
  • ② 데이터 아주 많을 때(수만 줄~)
  • ③ 집계·계산을 빠르게 (예: 요일별 평균)
  • 가벼운 혼자 기록 = 구글시트(4일차 강사 데모)로도 충분 · 앱 저장의 기본은 4일차처럼 DB

Supabase — 4일차에 만든 그 DB, 원리 복습

  • 웹에서 표(테이블) 생성 → supabase-py로 넣기(.insert())·꺼내기(.select()) — 4일차 앱이 한 일이 정확히 이 두 줄
② 개념 예시 (파이썬)
from supabase import create_client

# 프로젝트 주소와 키는 Supabase 대시보드에서 복사
supabase = create_client(URL, KEY)

# 데이터 한 줄 넣기
supabase.table("visits").insert({"city": "서울", "temp": 31.2}).execute()

# 데이터 꺼내기 (서울 것만)
rows = supabase.table("visits").select("*").eq("city", "서울").execute()
💡 RLS — DB의 잠금장치
  • RLS(Row Level Security) = 기본 전부 차단
  • 허용 규칙(정책) 켠 것만 보임
  • “데이터 안 보여요” = 고장 아님, 안전하게 잠긴 것
🔑 키 두 종류 — 절대 헷갈리지 말 것
  • publishable 키(예전 anon) = 앱에 넣어도 되는 공개용 (RLS가 보호)
  • secret 키(예전 service_role) = 모든 잠금 여는 마스터 → 웹앱·GitHub 업로드 금지
  • 비밀 금고(Secrets)에만 보관

Neon — SQL을 직접 쓰는 PostgreSQL

  • PostgreSQL을 무료로 켜 주는 서비스
  • 스트림릿 → st.connection(type="sql") 한 줄 연결 + SQL 직접 작성
② 개념 예시 (스트림릿 + SQL)
import streamlit as st

conn = st.connection("neon", type="sql")   # Neon 연결

# 도시별로 방문 수를 세어 많은 순으로 상위 5개
df = conn.query("""
    SELECT city, COUNT(*) AS n
    FROM visits
    GROUP BY city
    ORDER BY n DESC
    LIMIT 5
""")
st.bar_chart(df, x="city", y="n")
💡 SQL 단어 네 개면 집계 끝
  • COUNT(세기) · GROUP BY(묶기) · ORDER BY(정렬) · LIMIT(개수)
  • “도시별로 묶어 세고, 많은 순으로 5개만”
  • 이 SQL도 AI에게 시켜서 받으면 됨

2. 자동화 + 흥미로운 API

  • 사람이 매일 손으로 = 지침 → 컴퓨터가 정해진 시간마다 = 자동화
  • 세상엔 공짜 흥미로운 API 많음

GitHub Actions — 매시간 알아서 데이터 쌓기

  • GitHub이 정해진 시간마다 내 코드 대신 실행
  • .github/workflows/에 *.yml 하나 두면 됨
② 개념 예시 (.github/workflows/collect.yml)
name: 데이터 수집
on:
  schedule:
    - cron: "5 * * * *"      # 매시간 5분에 실행 (UTC 기준!)
  workflow_dispatch:          # 이 줄이 있으면 '수동 실행' 버튼도 생김
jobs:
  run:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: python collect.py
        env:
          API_KEY: ${{ secrets.API_KEY }}   # 비밀키는 Secrets에서
⏰ cron 시간은 UTC (한국시간 아님)
  • cron: "5 * * * *" = 매시간 5분에
  • 기준 = UTC · 한국시간 = +9시간
  • 한국 낮 12시 → UTC 새벽 3시(0 3 * * *)
💡 비밀키는 GitHub Secrets에
  • API 키 = 코드 직접 작성 금지 → Settings → Secrets and variables → Actions
  • 꺼내 쓰기 = ${{ secrets.API_KEY }}
  • workflow_dispatch → Actions 탭 수동 버튼으로 즉시 시험
🛰️ 실물 구경 — 이 수업의 데이터가 바로 이 방식으로 살아있습니다
  • 수업에서 쓴 예제 저장소(github.com/greatsong/modudata)의 Actions 탭을 열어 봅니다
  • 수업에서 배운 기온(4일차)·인구(2일차)를 포함해 박스오피스 매일 · 혼잡도 매시간 · 지하철 매월 — 로봇이 알아서 갱신 중 🤖
  • 4일차 기온 데이터(seoul.csv)가 늘 어제 기온까지 담고 있던 비밀 = 지금 배운 cron 그 자체
  • 위 개념 예시는 뼈대만 — 실물 .yml엔 라이브러리 설치와 결과 저장(커밋) 단계가 더 붙습니다 · Actions 탭에서 직접 확인
